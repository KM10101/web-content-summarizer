# 网页内容总结工具

## 项目定位

这是第一阶段的第一个候选主项目，用来承接本周的最小实验，并为下一周的 MVP 推进提供明确方向。

这个项目的目标不是一开始做成完整产品，而是先完成一个**可验证、可运行、可复盘**的最小版本。

## 项目目标

输入一个网页 URL，工具自动：

1. 获取网页内容
2. 提取主要正文
3. 调用 LLM 生成结构化总结
4. 返回适合阅读和后续处理的结果

## 为什么当前选择这个项目

这个方向适合当前阶段，主要因为它能够同时覆盖：

- 网页内容获取
- 正文提取与清洗
- Prompt 设计
- Structured output
- 最小 workflow 设计
- AI coding workflow 的实践与复盘

相比纯文本总结实验，它更接近真实应用；相比更复杂的 Agent 类项目，它的复杂度又相对可控。

## 当前阶段定位

当前阶段将这个项目视为：

- **本周的实验方向载体**
- **下一周 MVP 推进的主线候选**

也就是说，本周不追求把它做完整，而是先把方向、边界和最小链路定义清楚。

## 第一版 MVP 范围

### 要做的内容

第一版只包含以下能力：

1. **输入一个 URL**
   - 用户提供一个网页链接

2. **获取网页内容**
   - 抓取网页 HTML 或正文来源内容

3. **提取主要正文**
   - 尽量清洗掉导航、广告、无关噪音
   - 获取一份可以交给模型处理的主要文本

4. **生成结构化总结**
   - 调用 LLM 对正文进行总结
   - 返回稳定、可读、可继续处理的结构化结果

5. **提供最小运行方式**
   - 优先使用脚本、CLI 或最小 API 形式运行
   - 暂不以复杂前端页面为重点

## 第一版明确不做的内容

当前阶段明确不纳入范围：

- 用户系统
- 历史记录管理
- 登录鉴权
- 多模型切换
- 批量 URL 处理
- 浏览器插件
- 高级前端 UI
- RAG 能力
- Agent 化自动规划
- 收藏与知识库沉淀
- 登录态网页支持
- 对所有网站的完整兼容

## 输入输出定义

### 输入

- 一个网页 URL

### 输出

第一版建议使用如下结构：

```json
{
  "title": "",
  "summary": "",
  "keyPoints": [],
  "actionItems": [],
  "tags": []
}
```

### 字段说明

- `title`：文章标题
- `summary`：整体摘要，建议 2 到 4 句
- `keyPoints`：关键要点列表，建议 3 到 5 条
- `actionItems`：如果内容中存在明确可执行建议，则提取；没有则可为空
- `tags`：主题标签，建议 2 到 5 个

## 推荐工作流

第一版按固定 workflow 推进：

1. 输入 URL
2. 获取网页内容
3. 提取正文
4. 构造 prompt
5. 调用模型
6. 输出结构化总结

这个 workflow 的目标不是追求灵活，而是保证：

- 可控
- 可验证
- 可调试
- 可复盘

## 本周重点验证的问题

本周围绕这个项目，重点不是“把项目做完”，而是验证下面几个问题：

1. 网页正文提取是否足够容易落地
2. 结构化总结输出是否稳定
3. 总结结果是否具有基本可用性
4. 这条链路是否值得继续作为第一阶段主项目推进

## 本周成功标准

如果本周能满足下面这些条件，就说明这个方向值得继续：

- 已明确项目方向与 MVP 边界
- 已完成至少一条最小链路的实验验证
- 已能定义出稳定的结构化输出格式
- 已对该方向形成继续推进的判断
- 已把过程沉淀进每日记录与周复盘材料

## MVP 开发任务拆分

### 模块 1：项目骨架与运行入口

目标：先把项目初始化到一个可运行、可继续迭代的状态。

任务清单：

- [ ] 确定项目目录结构
- [ ] 确定运行方式（优先 CLI 或脚本入口）
- [ ] 建立最小 README 或运行说明
- [ ] 约定输入输出方式

完成标志：

- 能执行一个命令或脚本
- 能传入 URL 或固定测试输入
- 项目具备继续填充逻辑的基本骨架

### 模块 2：网页内容获取

目标：先把网页拿下来。

任务清单：

- [ ] 输入一个网页 URL
- [ ] 请求网页内容
- [ ] 返回原始 HTML 或基础文本
- [ ] 做最基础的错误处理

完成标志：

- 给一个 URL，程序能稳定拿到网页原始内容

### 模块 3：正文提取与清洗

目标：从网页中提取“可以拿去总结”的主要文本。

任务清单：

- [ ] 提取网页标题
- [ ] 提取正文主体
- [ ] 去掉明显无关内容
- [ ] 控制文本长度，避免内容过脏过长

完成标志：

- 给一个 URL，能得到标题 + 一份相对干净的正文文本

### 模块 4：LLM 总结链路

目标：把正文送给模型，并拿回结构化总结。

任务清单：

- [ ] 设计 prompt
- [ ] 定义输出 schema
- [ ] 调用模型
- [ ] 解析结构化结果
- [ ] 检查输出稳定性

完成标志：

- 至少对 1 到 3 篇网页内容，能稳定产出结构化总结

### 模块 5：结果输出与最小可用性

目标：让结果不只是“程序跑完了”，而是“可以看、可以用”。

任务清单：

- [ ] 终端打印格式化结果，或输出 JSON / Markdown 文件
- [ ] 让错误信息可读
- [ ] 视情况保留必要调试信息

完成标志：

- 输入 URL 后，能看到一份结构清晰的结果

### 模块 6：实验记录与项目收敛

目标：把这次实践沉淀成系统资产，而不是一次性动作。

任务清单：

- [ ] 记录实验输入
- [ ] 记录实验结果
- [ ] 记录问题与判断
- [ ] 记录哪些部分值得继续迭代
- [ ] 反推 MVP 边界是否合理

完成标志：

- 至少有一条实验记录
- daily 文档里能看出当天推进了什么
- 周末复盘时有材料可用

## 本周推进重点（周三到周五）

### 周三：打通最小链路

重点模块：

- 模块 1：项目骨架与运行入口

目标：

- 至少测试 2 到 3 个网页样本
- 能判断这个方向是否值得下周继续推进
- 项目 MVP 边界进一步明确
